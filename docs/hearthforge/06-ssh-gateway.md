# SSH Gateway

This document describes the Rust SSH gateway and why it acts as a jump host rather than replacing sshd in dev containers.

## Golden Path

Use this as the default workflow for new developers.

1. Admin provisions dev access:
   ```bash
   forge hearthforge add-dev
   ```
   Select project, provide developer public key. Command prints an SSH config snippet.

2. Developer pastes snippet into local `~/.ssh/config`.

3. Developer tests SSH in terminal:
   ```bash
   ssh <dev>-<project>
   ```

4. Developer opens VS Code / Cursor / JetBrains:
   - `Remote-SSH: Connect to Host...` → select `<dev>-<project>`
   - Open `/workspace/<project>`

> Do **not** use `ssh <dev>-<project>@ssh.dev.<domain> -p 2224` for an interactive shell. The gateway is transport-only and intentionally rejects session shells. Always connect through the final alias that uses `ProxyJump`.

## Why Jump Host Mode

VS Code / Cursor / JetBrains Remote-SSH expect:
- A normal SSH server endpoint
- SFTP support
- Ability to install and run a remote server component

To satisfy this, dev containers run OpenSSH (`sshd`). The gateway enforces authentication and routing but does not terminate developer sessions.

## Connection Flow

```
laptop
  └─ ssh <dev>-<project>
       └─ ProxyJump → gateway:2224 (authenticates as <dev>-<project>)
            └─ direct-tcpip → dev-<project>-<dev>:22 on dev-web
                  └─ sshd inside container (authenticates as dev)
                       └─ developer shell / Remote-SSH session
```

Developers never touch the host.

## Client SSH Config

The SSH config snippet generated by `forge hearthforge add-dev` has two parts: the gateway jump host entry and the container entry. The container entry includes `LocalForward` directives for every port defined in the project's `dev_ports` configuration, enabling automatic SSH port forwarding.

```sshconfig
Host <dev>-<project>-gw
  HostName ssh.<dev_base_domain>
  Port 2224
  User <dev>-<project>
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
  StrictHostKeyChecking accept-new

Host <dev>-<project>
  HostName dev-<project>-<dev>
  Port 22
  User dev
  ProxyJump <dev>-<project>-gw
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
  StrictHostKeyChecking accept-new
  # Dev server port forwarding — auto-generated from registry/projects.json
  LocalForward <frontend-port> localhost:<frontend-port>
  LocalForward <backend-port> localhost:<backend-port>
```

The `LocalForward` lines mean that when the developer runs `ssh <dev>-<project>`, all configured ports are automatically tunneled to their local machine. They can open `localhost:<frontend-port>` in their browser and access their running dev server with no extra flags and no proxy changes.

Example for developer `santiago` on project `tiap` (frontend: 3000, backend: 5000) with base domain `dev.qyvos.com`:

```sshconfig
Host santiago-tiap-gw
  HostName ssh.dev.qyvos.com
  Port 2224
  User santiago-tiap
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
  StrictHostKeyChecking accept-new

Host santiago-tiap
  HostName dev-tiap-santiago
  Port 22
  User dev
  ProxyJump santiago-tiap-gw
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
  StrictHostKeyChecking accept-new
  # Dev server port forwarding
  LocalForward 3000 localhost:3000
  LocalForward 5000 localhost:5000
```

With this config, `ssh santiago-tiap` gives a shell inside the container and automatically makes `localhost:3000` and `localhost:5000` available on the developer's local machine. Hot module replacement, WebSockets, and file watchers all work without modification. The ports are never exposed publicly — all traffic travels through the encrypted SSH tunnel.

## Key Setup (Windows / Cursor / VS Code on Windows)

```powershell
ssh-keygen -t ed25519 -C "you@example.com" -f $env:USERPROFILE\.ssh\forge_ed25519
Get-Content $env:USERPROFILE\.ssh\forge_ed25519.pub
```

Register the key:
```bash
forge hearthforge gateway-add-key --dev <dev> --pubkey "<full ssh-ed25519 ... line>"
```

Use `IdentityFile ~/.ssh/forge_ed25519` in `C:\Users\<user>\.ssh\config`.

## Key Setup (Linux / macOS / WSL)

```bash
ssh-keygen -t ed25519 -C "you@example.com" -f ~/.ssh/forge_ed25519
cat ~/.ssh/forge_ed25519.pub
```

Register the key:
```bash
forge hearthforge gateway-add-key --dev <dev> --pubkey "<full ssh-ed25519 ... line>"
```

## Identity and Policy Mapping

- Public key → developer identity
- Developer identity → allowed projects
- Allowed project → container target

Sources:
- `registry/devs.json` — source of truth for developer identities and access
- `gateway/authorized_keys/` — SSH public keys per developer
- `registry/gateway.toml` — gateway listen address, host key path, log directory

The gateway refreshes `devs.json` on each auth attempt — access changes apply without restarting the gateway.

## Developer Key Lifecycle

**Canonical store:** `/opt/infra/forge/gateway/authorized_keys/<dev>.pub`
- Each non-empty, non-comment line is treated as one allowed key for that developer

**On `forge hearthforge add-dev`:**
- Prompts for (or reuses) the developer's SSH public key
- Ensures the key is present in the canonical gateway store
- Writes the same key into `_keys/<project>/<dev>/dev` for the container

**On `forge hearthforge gateway-add-key`:**
- Adds additional keys for an existing developer (e.g. second laptop)
- Appends to the canonical gateway file
- Updates `_keys` for all projects that developer currently has

## Host Keys

Gateway host keys must be persistent:
- Store under `gateway/keys/`
- Do not generate ephemeral host keys on every boot

## Logging

Gateway logs at minimum:
- Timestamp
- Source IP
- Developer identity (or unknown)
- Auth success/fail
- Project selected
- Container target

```bash
# Recent gateway events
sudo docker logs proxy-gateway-1 --tail=80

# Persistent audit log
sudo tail -n 80 /opt/infra/forge/gateway/logs/audit.log

# Filter accepted/rejected auth
sudo rg "result=accepted|result=rejected" /opt/infra/forge/gateway/logs/audit.log
```

## Gateway Deployment

The gateway runs as a Docker container on the `dev-web` network.

### Build

```bash
cd /opt/infra/forge/hearthforge/gateway
sudo docker build -t forge-gateway:latest .
```

### Compose Service

```yaml
services:
  gateway:
    image: forge-gateway:latest
    restart: unless-stopped
    networks:
      - dev-web
    ports:
      - "2224:2224"
    volumes:
      - /opt/infra/forge/registry:/opt/infra/forge/registry:ro
      - /opt/infra/forge/hearthforge/gateway/keys:/opt/infra/forge/hearthforge/gateway/keys
      - /opt/infra/forge/hearthforge/gateway/authorized_keys:/opt/infra/forge/hearthforge/gateway/authorized_keys
      - /opt/infra/forge/hearthforge/gateway/logs:/opt/infra/forge/hearthforge/gateway/logs

networks:
  dev-web:
    external: true
```

### gateway.toml

```toml
[server]
listen_addr = "0.0.0.0:2224"
host_key_path = "/opt/infra/forge/hearthforge/gateway/keys/ssh_host_ed25519_key"

[paths]
devs_json = "/opt/infra/forge/registry/devs.json"
authorized_keys_dir = "/opt/infra/forge/hearthforge/gateway/authorized_keys"
audit_log_dir = "/opt/infra/forge/hearthforge/gateway/logs"
```

### Host Key Generation

```bash
sudo ssh-keygen -t ed25519 -N '' \
  -f /opt/infra/forge/hearthforge/gateway/keys/ssh_host_ed25519_key

# Fix ownership if gateway runs as uid 1000
sudo chown -R 1000:1000 /opt/infra/forge/hearthforge/gateway/keys
```

### Firewall and DNS

```bash
sudo ufw allow 2224/tcp
```

DNS: create `ssh.dev.<dev_base_domain>` pointing to the VPS public IP.

### Start and Verify

```bash
sudo docker compose up -d gateway

docker ps | grep gateway
sudo ss -lntp | grep 2224
sudo docker logs gateway --tail=50
```
